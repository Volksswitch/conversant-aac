## Project Vision
Build an AI-driven AAC system that gives non-speaking individuals access to **real-time conversation** — currently impossible with existing AAC devices. The core problem is not words-per-minute; it is the human aversion to awkward silence (~4 seconds). AAC users are locked into near-real-time or batch communication, which marginalizes them and restricts them to transactional rather than interactional communication.

**Mechanism:** Generative AI presents response options within seconds of a communication partner's utterance. The user selects one; the device speaks it. The system speaks *as* the user, not *for* them.

---

## Target User (Initial)
Literate, non-speaking individual with cerebral palsy (CP), 16+, using **direct select** (touch/pointer) as the access method. Vision is to expand to wider range of ages, literacy levels, and accessibility profiles over time.

## Architecture Components (from the paper)
1. **Situational Awareness** — GPS, face/voice recognition, calendar
2. **NLP In** — Speech-to-text
3. **Statement/Response Generation** — LLM (the linchpin)
4. **Statement/Response Shaping/Tuning** — worldview model, relationship modeling, conversation history
5. **NLP Out** — Text-to-speech

## Build Decisions (confirmed)

**Platform:** Static web app served from GitHub Pages (or similar). User points browser at a URL. App runs entirely in-browser. No backend server.

**Target hardware:** The File System Access API (FSA) is not supported on iOS or Android, and Apple does not make a tablet-class Mac. Therefore the device must be a Windows tablet running Microsoft Edge or Google Chrome. The Microsoft Surface is the primary candidate.

**Local data storage:** File System Access API — the app is granted permission to read/write a local folder on the user's machine. All worldview data, conversation history, and profile data stays on-device. No data sent to any server except the LLM API.

**Cross-device transfer:** Later feature — user manually exports a data package and moves it via iCloud, Google Drive, or OneDrive.

**AI vendor:** Claude API (Anthropic) first. Multi-vendor support added later. User creates their own Anthropic account and supplies their own API key. The Claude API is pay-as-you-go, billed per token — there is **no ongoing free tier for API access** (the free tier on the consumer claude.ai app does not apply to the developer API). The user pays only for the conversations they actually have. API key stored locally.

> **Vendor choice is a USER decision — do not architect around Anthropic only. (Ken, June 14 2026.)** Anthropic is the *first* implementation, not the *only* one. The design must keep the LLM provider swappable: which vendor (Anthropic, OpenAI, Google, a local model, …) and which model tier is ultimately the user's choice. **Architectural requirement going forward:** isolate all provider-specific concerns behind a provider-agnostic interface (a generation call that takes conversation + profile/system text and returns `{classification, options, missing_facts}`), so adding a vendor is a new adapter, not a rewrite. *Current Anthropic-specific touchpoints to abstract when we add the second vendor* (all in `app/js/llm.js`): the `api.anthropic.com/v1/messages` endpoint, the `x-api-key` / `anthropic-version` / `anthropic-dangerous-direct-browser-access` headers, the request body shape (`system` + `messages` with `user`/`assistant` roles), the model id string, and the `usage.input_tokens`/`output_tokens` response fields. Settings currently labels the field "Claude API Key" and the key validation assumes `sk-ant-…`; both become vendor-dependent. Pricing is per-vendor/per-model (`app/data/pricing.json` is currently single-rate). Keep this list current as the abstraction is built.

> **Future setting — quality-driven model tier (upgrade/downgrade). (Ken, June 14 2026.)** Add a user-facing control to move the model up or down a tier (e.g., within Anthropic: Haiku ↔ Sonnet ↔ Opus) based on the user's *perception of response-option quality* vs. cost. Today the model is hardcoded to `claude-sonnet-4-6` (chosen for the user-pays-per-token cost balance). This setting makes the intelligence ↔ cost tradeoff the user's call, surfaced in plain terms ("responses not good enough? try a stronger model — it costs more per conversation"). Pairs naturally with the multi-vendor abstraction above (tier is a per-vendor concept). Not built yet.

**Speech-to-text transcript validation:** Displaying the transcript of what the partner said is **critical** — not optional. The user must be able to confirm the system heard correctly before selecting a response.

**End-of-utterance detection (continuous partner capture):** A silence period (the "Optional Responses silence period" setting) is a *checkpoint, not a stop*. Each time the partner pauses for that period, the speech collected so far is sent to the AI for response options and a placeholder is spoken; recording continues. Resumed speech is appended and the combined utterance is re-sent after the next pause, so each checkpoint yields a more complete option set. Recording stops only when the user selects a response, at which point the transcript is cleaned once and the exchange is stored. A "latest-wins" generation token discards superseded in-flight requests, and a persistent "Please repeat what you said." control discards a garbled capture and keeps listening. An **auto-resume-listening** setting (Settings → General; default **off**) turns partner recording back on automatically after a response is spoken, so the user need not tap Start Listening between exchanges. (Implemented June 2026; see `Continuous-Partner-Capture.docx` and Architecture Overview §9. **Refined in v0.2.32 (see Further Design Thoughts #4):** Start Listening never fires at startup, must be invoked manually at least once per session, and auto-resume is gated behind that first manual invocation (`manualListenArmed`), re-armed only after another manual Start once listening is manually turned off. This is the Phase 1 realization of the COMPLETE/INCOMPLETE/CONTINUING classifier and filler ladder above.)

**Voice/TTS:** Browser built-in TTS acceptable for MVP. Voice banking (personalized/cloned voice) is a valuable future feature.

**Communication partner:** Needs nothing. Entirely unaware of system internals. AR glasses are a future hardware form factor.

## Response Option Display

Response options shown to the user may appear in **compressed form** — a few words representing a fuller thought — to speed up reading. The system may also read the compressed hint aloud. The **trigger for the auditory hint is a user preference**: options include auto-playback as each option is highlighted in sequence, or on first tap (before a double-tap confirms selection). Additional triggers may be supported based on user feedback.

The user normally hears the **fully expanded response only when it is spoken** after selection. Whether the expanded text is displayed or read before confirmation may become a configurable option based on user feedback.

Position 1 in the options list is the AI's best guess. There is no special visual treatment to distinguish it — the design goal is simply that the first option is the best option, with that accuracy improving over time.

## Worldview Model
- Minimum viable version: a carefully selected "getting to know you" questionnaire (~100 questions)
- Questions are completed entirely at the user's own pace; none are required
- The system provides meaningful functionality even with zero worldview questions answered
- Info gathering must be **chunked** — physically draining for CP users
- Many real-time exchanges need zero worldview context ("Hi, how are you?" needs no personalization)
- RAG will reduce token overhead as the profile grows over time
- Designing the specific questionnaire content is part of the work ahead

## Rollout Sequence

**Phase 1 — Core Conversation Loop (Proof of Concept)**
STT → display transcript → placeholder utterance auto-spoken → LLM generates response options → user selects → TTS speaks full response. No worldview shaping, no situational awareness.

Key Phase 1 features:
- **3 response options** by default (configurable)
- **Conversation context:** rolling window of the current session only
- **Placeholder/filler utterances** (e.g., "Just a second…", "Let me think about that") automatically generated and spoken while the user is selecting; customization supported
- **Conversation initiation:** via a set of canned conversation starters (pre-configured phrases the user can select to open a conversation); later refined by situational awareness in Phase 2
- **Compressed display / expanded speech** for response options (see above)
- **Identity / personalization — now the worldview profile (was the interim name + "About You" seed):** The June-2026 stopgap (Settings → General "Your Name" / "About You" fields injected into the prompt) has been **removed in worldview Build Step 4**. Personalization now comes entirely from the worldview questionnaire ("About Me"), injected via `worldview.buildBlock()`. The no-bracket safety rule still always applies (the prompt forbids `[Name]`/`[city]` and tells the model to phrase around any unknown detail), so the app degrades gracefully with an empty profile. See the **Worldview Questionnaire (June 2026)** section.
- **Offline review/feedback mode:** deferred — too complex even in rudimentary form
- **Response option UI layout:** requires a design discussion on how much of the screen is allocated to traditional AAC vocabulary vs. AI-facilitated response options; ultimately user-configurable

**Phase 2 — Situational Awareness**
Add location context (GPS), communication partner recognition (face/voice), and calendar integration. Canned conversation starters become contextually suggested based on location and partner identity.

**Phase 3 — Worldview Shaping and Tuning**
Integrate the worldview model. Responses shaped to reflect the individual user's personality and goals. RAG employed to manage profile size. Offline feedback/review mode introduced here.

## Advanced Future Capability: Streaming Response Generation
The long-term vision is for the system to begin generating and refining response options *while the communication partner is still speaking* — using partial STT transcripts as they arrive. Options would be continuously updated and narrowed as the utterance becomes clearer. This eliminates virtually all post-utterance latency. This capability requires significant design thought and is not targeted for Phase 1 or 2.

### Settled decisions — Conversation Engine

- **Sequence stack, not turn alternation**: conversation state is a stack of open sequences (adjacency pairs); repair sequences nest on top; the original obligation is restored automatically when they resolve.
- **Four-slot move palette**: PREFERRED / DISPREFERRED (hedge + account) / INITIATIVE / REPAIR — structurally distinct conversational moves, positionally stable for motor automaticity. (Refines the earlier "3 response options" default: the palette is typed by move, not just counted; option count remains configurable.)
- **Five modes**: LISTENING, RESPONDING, REPAIR-OF-SELF, INITIATING, PRE-CLOSING/CLOSING.
- **Access method independent of conversation logic**: engine emits a typed, prioritized palette descriptor; renderers (direct select now; scanning/eye gaze later) present it. Consistent with the direct-select initial target while keeping the expansion path clean.
- **System infers mode, user overrides** via persistent buttons (default set: Say again / Hold on / Pardon? / Wind down).
- **Single combined LLM call** classifies the partner's action (FPP type) AND generates options; classification is emitted first in structured output so it is inspectable. LLM stateless; app injects state.
- **Filler ladder** (refines the placeholder mechanism and the old 4 s initial delay): rung 1 acknowledgment token ≤1 s after end-of-utterance (no LLM — this is what makes transcript validation affordable), rung 2 projection filler ~2.5 s covering the confirm window, rung 3 periodic re-fill. CA: silence becomes meaningful past ~1 s.
- **Backchannels/continuers deferred**; end-of-utterance classification schema reserves COMPLETE | INCOMPLETE | CONTINUING so they slot in later without rework.
- **Highest-leverage near-term item**: REPAIR-OF-SELF mode — partner says "What?" → re-speak / rephrase / expand the user's last utterance.
- **Mode vs. floor are distinct dimensions; LISTENING is the system's *receptive posture*, not "the partner owns the floor" (refinement, Ken, June 15 2026, v0.3.2).** Ken's catch: the diagnostic panel showed `Mode: LISTENING` before any capture, and "partner has the floor" is wrong for the at-rest case. Two things were conflated. **`mode`** = what the *system* is doing (LISTENING / RESPONDING / REPAIR-OF-SELF / INITIATING / PRE-CLOSING-CLOSING). **`floor`** = whose turn it is — **open** (at rest / between turns, up for grabs), **partner** (partner producing or owed the next turn), **self** (user's turn to act). LISTENING spans *both* floor=partner (partner mid-utterance) and floor=open (startup, or the beat after a response) — so floor cannot be read off the mode, and the design doc's gloss of LISTENING as *"partner is speaking"* only covers the partner-mid-utterance case. Also: because the persistent overrides (Start conversation / Say again / Hold on / Pardon? / Wind down / "Please repeat" / the composer) are available in **every** mode, the user can seize the floor at any time — floor=partner is a *default expectation*, not an exclusive lock. **Built:** `floor` is a first-class field on the engine `ConversationState` + snapshot (`engine.FLOOR = {OPEN, PARTNER, SELF}`), set at each transition (partner capture → partner; INCOMPLETE → partner; COMPLETE → self; repair-initiator → self; selectMove/completeRepairOfSelf → open; pardon → partner; windDown/initiate → self), and shown as its own **Floor** row in the degenerate panel (open=grey, partner=orange, self=blue). The earlier **Capture (mic)** row (v0.3.1) stays — it's an I/O state, also distinct from both mode and floor (a silent-but-listening mic is still floor=open). **Mode shows "STANDBY" when genuinely at rest (v0.3.3, Ken).** Ken's follow-up: "LISTENING" reads wrong when the mic is off. Rejected relabeling Mode off the mic directly (re-couples the dimensions we just split) and the word "Holding" (collides with *floor-holding* — the filler ladder + the "Hold on" button). Instead the renderer shows **STANDBY** in the Mode cell only when the engine is *truly idle* — `capture off AND floor open AND empty stack AND no palette` — otherwise the engine's mode verbatim. **Presentation-only:** the engine's internal mode stays `MODE.LISTENING` (no sixth mode, engine untouched). Because the label now depends on both the snapshot *and* the mic state, `ui.js` keeps the last snapshot + a `capturing` flag and re-renders the Mode cell from `renderModeCell()` whenever either changes (so it flips STANDBY↔LISTENING the instant the mic toggles, not only on the next snapshot). *(Open question deferred: floor transitions are best-effort heuristics in the engine; whether the renderer should ever override them, and how lapses are handled, is unspecified.)*
- **No transcript-confirmation gate — generation fires on the silence period, not on a confirm tap (DECIDED Ken, June 15 2026).** The Conversation-Engine-Design.docx §6 line *"the generation window starts only after the user confirms (or corrects) the transcript"* — the CA-purist intersubjectivity gate — is **relaxed to a configurable default, off.** Generation starts after `n` seconds of partner silence (the existing "Optional Responses silence period" setting), with **no** explicit transcript-confirm step, to shorten the user's time-to-respond. This is not a new departure: (a) the design doc already lists **"confirmation gates"** among the *defaults of the standard profile, not fixed rules* — it is **not** an invariant (the invariants are repair path / floor-holding / closing / free composition / visibility of system speech / revertibility); and (b) the shipped **Continuous Partner Capture** already generates on each silence checkpoint with no confirm tap. **What we keep (these are what make "no gate" safe):** (1) the **transcript still displays** (CLAUDE.md "critical, not optional") — the user *sees* what was heard, just isn't required to tap-confirm; (2) the **REPAIR / "Pardon?" slot + persistent "Please repeat what you said" control** is the recovery for STT errors — the old gate caught errors *before* generation, the relaxed model catches them *after* (cheaper in time; repair is an invariant); (3) the auto-confirm-above-confidence-threshold knob was already the escape hatch — this effectively sets it to "always." The three-state transcript display may survive as **informational** (heard / generating / ready) but no longer **blocks** generation. **Setting:** "Require transcript confirmation before generating" — Profile-tier, default **off**, for the rare CA-purist user. **Docs to update (deferred):** `Conversation-Engine-Design.docx` §6 and `UI-Design.docx` transcript-gate section must stop asserting the gate as mandatory.

### Settled decisions — UI

- Four screen regions: transcript (with mode chip), move palette, persistent controls, composer access. Geometry never reflows on mode change — "the cards changed," never "the screen changed."
- Move card anatomy: glanceable hint (primary reading target; names the action, not a truncation), full TTS text, slot badge, format tag, latency dot (filled = instant, hollow = generation round-trip).
- **Triple coding**: slot identity = position + color + text badge; never color alone. Defaults: green preferred, amber dispreferred, blue initiative, purple repair.
- Transcript is a three-state *display*: amber UNCONFIRMED → blue CONFIRMED·GENERATING → green OPTIONS READY. **It no longer gates generation by default** — per the Conversation-Engine decision above (Ken, June 15 2026), generation fires on the silence period, not on a confirm tap; the states are informational. The blocking gate (never generate from an unconfirmed transcript) survives only as a Profile-tier setting, default off.
- **Nothing is spoken on the user's behalf invisibly** — every filler/token is displayed as it plays.
- **Palette updates queue until a selection boundary**: options never change under a user mid-selection (mandatory for future scanning/dwell renderers).
- Engine→UI contract is a versioned palette-descriptor JSON with `priority` and `droppable` per move; renderers drop/page by priority, never by move type.

### Configurability philosophy (overarching)

- **As many decisions as possible are user-determined and configurable.** This population is made up of unique individuals; the user experiments with a flow and adjusts from experience. Behavioral statements in the design docs are *defaults of the standard profile*, not rules.
- **Invariants are guaranteed capabilities, not fixed behaviors**: a repair path, a floor-holding path, a closing path, free composition, visibility of system speech, and revertibility must always exist; how they manifest is configurable. Config validation enforces "at least one path" rules at save time, with explanation — never silent correction.
- Three settings tiers by *when* a change is safe — Quick (mid-conversation) / Profile (between conversations) / Setup (rare, may be supporter-assisted) — all user-owned; nothing locked away from the user.
- Experimentation loop: named profiles (bindable to Practice Mode scenarios), automatic change journal, session review metrics (descriptive, local, private — inform the user, never score them), guided trials, one-tap revert. Revertibility is the one absolute.
- **Confirm before destroying significant work (standing rule, Ken, June 15 2026).** *Every* action that can wipe away significant user work — clearing/restarting answers, deleting people or relationships, bulk resets, overwriting a saved profile — MUST include an explicit validation step before it runs. This is a build checklist item, not an afterthought: when adding any clear/reset/delete/overwrite control, wire its confirmation in the same change. (Trivial, easily-redone actions — a single-field edit/skip with Undo, resetting the usage-token counter — don't require it; the bar is "significant work.")
  - **The confirmation must NOT be the native `window.confirm()` (Ken, June 15 2026, v0.2.30).** Native confirm renders a generic dialog indistinguishable from routine browser/OS prompts — notably the File System Access folder-permission popup — so the user dismisses it on autopilot and a destructive action doesn't read as dangerous. Use the **`confirmDanger()`** modal in [confirm-dialog.js](app/js/confirm-dialog.js) instead: a deliberately distinct red danger card (⚠ mark, red title + red confirm button, `.danger-dialog` CSS), Cancel focused by default (a stray Enter cancels), `showModal()` top-layer so it sits above the full-screen About Me overlay. Returns `Promise<boolean>`; use `if (!(await confirmDanger({title, body, confirmLabel, cancelLabel}))) return;`.
  - *Audit June 15 2026:* About Me **Restart** (`onRestart`) and **Remove person** route through `confirmDanger`; the only un-confirmed destructive action is the Settings usage-counter reset, which is below the bar.

### Test-suite Pass 1 — filler timing + TTS-echo feedback loop (Ken, June 16 2026), v0.3.4

Two defects Ken found running the live conversation-engine test suite, with **Ken's chosen approach** for the second:

1. **Placeholders started/repeated too soon — now driven by the Settings values.** The §6 filler ladder used fixed constants (rung 1 at 0.8 s, rung 2 at +1.7 s) and ignored the user's two Settings: "Initial Placeholder Statement Delay" (`initialDelay`, default 4 s) and "Subsequent Placeholder Statement Delay" (`subsequentDelay`, default 10 s) — exactly the controls whose hint says they govern this. Rewrote `placeholders.js`:
   - **First placeholder lands `initialDelay` seconds after the partner *finishes* speaking** (matching the Settings hint). Because the silence checkpoint that triggers the ladder already consumed `silencePeriod` (`loadSilenceThreshold()`) seconds of that wait, rung 1 waits the remainder: `max(MIN_RUNG1_DELAY=500ms, (initialDelay − silencePeriod) × 1000)`. Defaults → 2 s after the checkpoint = **4 s after the partner stops**.
   - **Subsequent placeholders re-fill every `subsequentDelay` seconds** (rung 1 → next filler is now one `subsequentDelay`, not 1.7 s; the old `rung2`/`rung3` collapsed into one recurring `refill()`/`scheduleRefill()`). Rung 1 is still the inline ACK token (no fetch); rung 2+ are the projection fillers from `placeholders.json`, never the same twice in a row.

2. **App's own TTS was captured by the mic as partner speech (the feedback loop).** While the filler ladder spoke, the mic stayed live (`listeningIntent` true), so SpeechRecognition transcribed the placeholder audio, appended it to `accumulatedText`, reset the silence timer, and **re-fired `handleSilencePeriod`** — which restarted the ladder at rung 1 ("begin/repeat too soon") and bumped `generationToken`, **delaying and even abandoning the already-generated responses** in favor of a regeneration from `partner statement + filler comments`. **Ken's approach (chosen over muting the mic): the app knows exactly what it spoke, so discard only that content — the mic stays on so a partner talking over the filler is still heard; only *unique* content renews the partner's turn.** Implemented as content-based echo filtering, not mic gating:
   - **`tts.js`** broadcasts speaking-state with the spoken text: `onSpeakingChange(cb)` fires `cb(true, text)` on each `speak()` start and `cb(false, null)` on end; a monotonic `speakToken` stops a superseded utterance's late `onend` from reporting a spurious end. `cancel()` also reports the end. Added `isSpeaking()`.
   - **`app.js`** wires it: `tts.onSpeakingChange((speaking, text) => speaking ? stt.noteSpokenStart(text) : stt.noteSpokenEnd())`.
   - **`stt.js`** keeps a list of `activePhrases` the app is currently speaking (each stays matchable for `ECHO_TAIL_MS = 1500 ms` past end-of-speech, to cover recognition lag/trailing audio). `onresult` runs each captured segment through `isEcho()` and **drops matches** (doesn't accumulate, and — crucially — only calls `resetSilenceTimer()` when a *non-echo* segment was heard, so pure echo never renews the turn or re-fires generation). `isEcho` matching (normalized lowercase, punctuation→spaces): **exact** (segment is the phrase), **prefix** (segment is a leading slice — the recognizer's interim builds up as a growing prefix), and **embed** (phrase appears inside the segment — recognizer padding/merging) but **embed is restricted to multi-word phrases** so a short common ACK token ("Okay.", "Right.") can't swallow real partner speech that merely contains that word (e.g. "okay so what do you think"). Empty-after-normalize segments (pure punctuation/noise) count as echo.
   - **Accepted tradeoff:** if the recognizer merges a multi-word filler verbatim with the partner's resumed words into one segment, the whole segment is dropped (the filler is far more likely to be our echo the recognizer extended than the partner reciting our 4-word filler). In practice the filler and the partner's resumed speech are separated by a pause and arrive as different segments.

### Single-instance enforcement — design question (Ken, June 16 2026; my recommendation, NOT built)

Ken asked whether we can prevent multiple instances/launches of the app. **Yes, and it matters for this app specifically:** two live instances (two tabs/windows) each run their own mic + TTS, so instance A's spoken placeholder is captured by instance B's mic — **cross-instance feedback the v0.3.4 content-echo filter cannot catch** (B doesn't know what A said). Two instances also race on the FSA data-folder writes (`worldview.json` / `relationships.json` / conversation logs) and can double-speak. So single-instance is a robustness requirement, not just polish.

**Recommended approach (two layers, covers the realistic cases on a dedicated Surface in one browser):**
1. **Web Locks API (`navigator.locks.request`) for detection + a blocking "already open" screen.** On startup request an exclusive named lock (e.g. `aac-single-instance`) held for the page's lifetime via a never-resolving promise. The first instance gets it; later instances' requests stay pending → they detect another instance holds it and show a full-screen "AAC is already running in another window — switch to it" overlay instead of starting the mic/engine. Well-supported in Edge/Chrome (Chromium). Covers multiple tabs *and* windows in the same browser profile.
2. **PWA `launch_handler: { client_mode: "focus-existing" }` in the manifest** so re-*launching* the installed app focuses the running window rather than spawning a new one (the app is intended to be installed app-style on the Surface). Complements layer 1, which still backstops a manually-opened second tab.

**Boundary / limitation:** web APIs can't coordinate across *different* browser profiles or *different* browsers (Edge AND Chrome both open) — they don't share locks/localStorage/BroadcastChannel. On a dedicated device using one browser this is an acceptable boundary; if it ever matters, an FSA folder lock-file (write a PID/heartbeat into the data folder, stale-timeout to recover from crashes) is the only cross-browser option and is heavier/more fragile. *Not built — awaiting Ken's go-ahead. When built: bump version, add to the privacy/consent behavior already in flight, and note the cross-browser boundary in the docs.*

**Future: partner voice recognition as an extraneous-speech filter (Ken, June 16 2026).** When Phase-2 partner recognition lands, real-time speaker identification could filter out speech that isn't the recognized communication partner (the app's own TTS, bystanders, background voices) — a more general solution than content-matching the app's own fillers. Records alongside the v0.3.4 echo filter as the eventual upgrade path; the content filter is the no-extra-infrastructure stopgap until then. **[→ Overview To Do — Phase 2 partner recognition / privacy section]**

**Prompt caching — never pad to reach the cache floor; capture caching by prefix structure instead (decided Ken, June 15 2026).** Ken asked whether padding a prompt to the cache minimum is worthwhile. **No.** (1) The minimum is in **tokens, not characters**, and is **model-specific**: **2048 tokens** for Sonnet 4.6 / Fable 5, **4096 tokens** for Opus 4.x, 1024 for Sonnet 4.5 and older. ### Settled decisions

- **The keyboard is the easy part; voice + return-to-UI are essentially free.** Any keyboard (OS or in-app) just deposits text into a focused field in the page. The app then speaks it via the existing `tts.speak()` path (already uses the app-selected voice), and the app UI was never actually left. No integration risk on the "speak in the app's voice" / "return to the app" requirements regardless of keyboard choice.

- **OS / browser keyboard cannot be reliably summoned, and is a black box — so it is a *fallback*, not the foundation.** Technical reality on a Windows tablet (Edge/Chrome):
  - There is **no web API to programmatically launch the Windows touch keyboard.** It auto-appears only when an editable element (`input`/`textarea`/`contenteditable`) receives focus *and* the device is in a touch context. "Invoke from a button" therefore means "focus a text field and hope the OS pops the keyboard."
  - On a Surface with the type cover attached (laptop mode) it often won't auto-pop at all. Behavior varies by tablet mode, Windows version, and browser.
  - The Chromium **VirtualKeyboard API** (`navigator.virtualKeyboard.show()`) only manages an already-available keyboard's overlay; it still requires focus on an editable element. Not a reliable on-demand "open system keyboard" switch.
  - It gives **zero control** over layout, key size, spacing, prediction, or access method — exactly the things that matter for a CP user's motor accessibility.

- **The real path is an in-app virtual keyboard.** It is the only option that:
  1. **Honors access-method independence** (engine emits a palette descriptor; renderers present it — direct select now, scanning / eye-gaze later). An OS keyboard can't participate in the future scanning renderer; an in-app keyboard *is* just another renderer.
  2. **Unlocks AI-assisted composition** — type a few words → expand into a full sentence *in the user's voice* (worldview model); personalized word/phrase prediction. The OS keyboard can touch none of this.
  3. Keeps visual + behavioral consistency (key sizing, triple-coding, dark mode, latency dots).

## Auto-update on launch / Start (June 14 2026), v0.2.10

**Problem (Ken):** the app didn't pick up new deployments on launch or when Start was pressed — the old "Ctrl+Shift+R dance." Two causes: (a) the network-first worker's `fetch(request)` still hit the **browser HTTP cache** (GitHub Pages serves `Cache-Control: max-age=600`), so a launch within 10 min of a deploy got stale assets; (b) the registration never rolled a redeployed worker into the running page — a new `sw.js` installed but the page kept the version it launched with.

**Fix (three parts):**
1. **`sw.js` revalidates.** Runtime fetch is `fetch(new Request(request, { cache: 'no-cache' }))` (ETag revalidation — can't serve stale within the max-age window). Precache `addAll` uses `new Request(u, { cache: 'reload' })` so the precached shell is fresh too. (`self.skipWaiting()` on install + `clients.claim()` on activate were already there.)
2. **Auto-reload on new worker (`index.html`).** A `controllerchange` listener reloads the page once when a freshly-activated worker takes control. Guarded with `hadController` (skip the first-ever install, which has no prior controller) and a `reloading` flag (no loop). `reg.update()` is called on `load` to check for a new worker every launch.
3. **Update check on Start (`app.js` `handleStart`).** `navigator.serviceWorker.getRegistration().then(r => r && r.update())` — a cheap no-op when nothing's new; if a new worker is found it activates and the `controllerchange` handler reloads. Covers the "left open, redeploy, press Start" case.

APP_VERSION / CACHE_VERSION → `0.2.10`. **Verified in preview:** worker registers + activates + controls the page, version 0.2.10 in About, no console errors, **no reload loop** on localhost (controllerchange doesn't fire after the first claim when nothing's new). *The redeploy→auto-reload path itself can't be exercised in the preview (needs a real new deployment) — confirm on the test tablet that a fresh deploy now appears on relaunch without a hard refresh.*

## Conversation & Relationship Goals — three-layer model (design recorded June 15 2026; NOT built)

**Ken's insight (June 15 2026):** everyone has goals for every conversation — consciously or not — from "maintain the connection with this person" to "steer the conversation toward X." Goals should be a first-class input to generation. The key framing: **"relationship goals" is really *three distinct things on three time horizons*,** and they map to three different stores — lumping them together is what makes it hard.

**The three layers:**
1. **Disposition** — how you generally are with people (durable, trait-level): "I keep things light," "I want to be seen as capable." Not tied to one person or conversation; the *prior/default stance*.
2. **Standing relationship goal** — what you want from a *specific relationship* (durable, person-level): "stay close to my sister," "I want my boss to see me as competent." Stable across many conversations with that person.
3. **Conversation goal** — what you want from *this exchange right now* (ephemeral, session-level): "today I need to ask Dad for help," "steer toward planning the trip." Set at/just-before conversation start, injected into that conversation's prompt, then discarded.

**Mapping to existing stores (two of three already have a home):**
| Layer | Store | Notes |
|---|---|---|
| Disposition | `worldview.json` **Tier-B** (how you treat different people, register/values) | Exactly what Tier-B is for; designed, not yet built. |
| Standing relationship goal | `relationships.json` **edge `attrs`** (e.g. `attrs.goals`) | **Clean fit today, no schema change** — a goal is an attribute of the `me→person` edge. |
| Conversation goal | **Session runtime** (Conversation Engine state), optionally appended to the `conversations/*.json` log for Phase-3 review | Genuinely new, but it's runtime state, **not a new persistent store**. |

**Recording mechanism (per layer):**
- Standing relationship goal → a **"what I want from this relationship"** control in the **People editor** (the one built in v0.2.27), writing to the edge `attrs`.
- Conversation goal → set at **conversation initiation** (pairs with Phase-1 canned starters and Phase-2 partner recognition — "you're talking to Dad; what do you want from this?"), with a quick mid-conversation control (fits the persistent-controls region); stored in session memory.
- Disposition → Tier-B questionnaire when built.
- **Expression format — DECIDED (Ken, June 15 2026): curated menu + free text.** A short list of common goals (stay close / be understood / be seen as capable / keep it light / get support / repair things) **plus** an "in my own words" option — fast to pick (matters for the CP target user) yet still personal. (Not menu-only, not free-text-only.)

**Why it matters / how it's used:** goals shape and **rank the four-slot move palette** (PREFERRED / DISPREFERRED / INITIATIVE / REPAIR): "maintain connection" → favor affiliative PREFERRED; "steer toward X" → surface a high-ranked INITIATIVE toward X; "wind down" → favor PRE-CLOSING. They become explicit generation inputs, **layered most-specific-wins** exactly like the prompt blocks already are: disposition (worldview) → standing relationship goal (graph) → conversation goal (session).

**Nice property:** goals are **never spoken** — they only steer generation — so they sidestep the privacy machinery entirely (unlike facts, a goal can't be accidentally said aloud).

**Open sub-questions (not yet decided — pin before building, so they aren't silently assumed):**
1. **Single vs. multiple goals per layer.** The mapping sketches `attrs.goals` (plural), but we only decided single-*relationship* (the relationship picker). Whether a relationship or a conversation can hold more than one goal at once is undecided. (Parallels the single-relationship-per-person call.)
2. **User-set vs. system-suggested goals.** Everything here assumes the user sets the goal explicitly. Whether the system may ever *infer/suggest* a goal (from partner recognition, history, etc.) is open — current default is explicit/user-set only.
3. **Per-person vs. relationship-type goals.** "With authority figures I want to seem competent" is a goal on a relationship *type*, not one person. The graph stores `type` on the edge, not as its own node, so type-level goals would need a convention (or a small types table). For now goals would attach per-person only.

**Status:** design recorded; expression format decided (curated menu + free text); three sub-questions above left open. **Build deferred** (Ken: "just record the design") — revisit when the conversation-engine work matures. Standing relationship goals are the smallest first build when we resume (they slot straight onto the existing graph edge + People editor).

## Further Design Thoughts (June 15 2026) — recorded, not built

A batch of Ken's thoughts captured this session. **Status: recorded only — none are implemented.** Items tagged **[→ Overview To Do]** also need to land in the Architecture and/or Product Overview docs; those edits are deferred and listed in the **Overview-Document To Do List** section below (Ken: don't spend editing time on the Overview docs for a couple of thoughts — let them accumulate and batch).

1. **Multi-party conversations — future feature.** Today the model assumes one communication partner. Supporting more than one partner in a conversation (knowing who said what, who a response is directed to, turn-taking among several people) is a future capability, not Phase 1/2. Touches STT (speaker diarization), the transcript UI, partner recognition (Phase 2), and the sequence-stack engine. **[→ Overview To Do]**

2. **Conversational honesty — the user is a person, not an encyclopedia (design principle).** The system must make it natural for the user to say **"I don't know,"** **"I think X but I could be wrong,"** or to state a sincere belief that happens to be wrong. It must **not** turn the user into a know-it-all / information oracle / smart speaker that others can interrogate or exploit. Rationale: the system speaks *as* the user, not *for* them — it mirrors the user's actual knowledge state and fallibility, it does not optimize for being maximally helpful/correct like an assistant. Concrete implications: the move palette should be able to surface "I don't know" and hedged/uncertain framings (good fit with the existing **DISPREFERRED = hedge + account** slot); the generation system prompt must steer toward authentic, fallible self-expression and away from assistant-style omniscience. This is a guardrail against the AAC user being treated like Google. **[→ Overview To Do]**

3. **"What this system is NOT" — add an explicit section (content below).** The system is **not** a personal assistant and has no intention of becoming one. It does not: review or triage your email, make appointments or manage your calendar, take notes in class, prepare study materials / sample tests, do research tasks, or otherwise act as a productivity agent. It is a **conversation** tool — it helps a non-speaking person hold real-time, interactional conversation *as themselves*. Scoping this explicitly protects the product's focus and sets partner/supporter expectations (it is not a smart speaker, not Siri/Alexa, not an LLM chatbot bolted to a tablet). **[→ Overview To Do — this is primarily a Product Overview positioning section]**

4. **"Start Listening" must be a deliberate, per-session user action — never automatic at startup. BUILT (June 15 2026), v0.2.32.** *Implementation:* `app.js` `manualListenArmed` session flag — set `true` by a manual start in `toggleListening`, set `false` by a manual stop; the automatic stop when a response is selected does **not** clear it. `handleResponseSelected` auto-resumes only when `manualListenArmed && storage.loadAutoRelisten()`. Startup was already clean (`handleStart()` never auto-listens). APP_VERSION/CACHE_VERSION → `0.2.32`. Rules satisfied:
   - Start Listening **never** takes effect simply because the app launched/started. Microphone capture begins only when the user explicitly invokes Start Listening.
   - The user must invoke Start Listening **at least once during a session** before any automatic listening can occur.
   - **Auto-resume** (the existing `auto-resume-listening` setting) is gated behind that first manual invocation: it only resumes listening after a spoken response *if* the user has manually started listening at least once this session.
   - If the user **turns listening off** during a session, the gate resets — auto-resume will not fire again until the user manually invokes Start Listening once more.
   **[→ Overview To Do — privacy/consent behavior belongs in both overviews]**

5. **Recording indicator for the communication partner (privacy — open question, my recommendation: yes).** Question Ken raised: do we need a visible UI/hardware indication that recording is taking place, primarily for the partner's benefit? **My take: yes, recommended and arguably ethically/legally prudent.** Points: (a) the partner is "unaware of system internals" by design, but a mic-active indicator reveals *no* internals — it only signals capture, which is a different thing; (b) audio-capture consent varies by jurisdiction (one-party vs. two-party/all-party consent recording laws), so a clear "listening" indicator is good practice; (c) **we don't store partner audio** — STT transcribes and the audio is discarded — but the *transcript* is stored, so "listening" is the honest signal; (d) on a Surface tablet we control only on-screen UI, so the near-term answer is a **prominent on-screen recording indicator**; a hardware capture LED (like a webcam light) is the gold standard but only relevant to a future dedicated-hardware / AR-glasses form factor. Pairs with item 4 (deliberate Start Listening) — together they make capture visible and intentional. **[→ Overview To Do]**

6. **Age 16+ target — rationale + an appropriateness filter (open question, my recommendation below).** Ken isn't certain why he set 16+; candidate reasons: (a) scarcity of free youth/preteen TTS voices; (b) concern the AI might suggest responses inappropriate for under-16s. On the filter: **yes, an appropriateness filter is achievable.** Layered options, cheapest first: (1) a **system-prompt register instruction** in `buildProfileBlock()` (set the appropriateness band); (2) an explicit content guard on generated options; (3) provider-side safety settings. **Where to set it so it can't be trivially flipped:** Ken's instinct (a password/PIN-protected Settings area) is right — it fits the existing **Setup-tier, supporter-assisted** settings concept. **Important framing:** the system speaks *as the user*; an adult user gets to choose their own register (even a coarse one) — it's their voice. So the appropriateness filter is really for **minors and supporter-managed accounts**, not a global nanny on every user. Design it as an **optional, supporter-locked profile setting** (default off for adult self-managed use; on/locked for minor accounts), not an always-on filter. **[→ Overview To Do]**

7. **Around RAG (Phase 3), accept user-authored documents representing the user's views.** When RAG support lands, let the user feed in documents (essays, prior writing, position statements, faith/values statements, etc.) that capture their views on various topics. These become retrievable worldview context — a richer, user-authored complement to the structured questionnaire. Fits the worldview/RAG progression already recorded (heuristic → client-side embeddings → vector store) and Phase 3's worldview shaping. **[→ Overview To Do — worldview/RAG section]**

8. **System requirements — be able to recommend CPU/disk specs; understand what onboard AI adds (my analysis).** Two distinct regimes:
   - **Current cloud architecture (Claude/other API):** requirements are **modest** — any modern Windows tablet running current Edge/Chrome, a few hundred MB for the cached app + local data folder, and a network connection. It introduces: a capable **GPU or NPU** with substantial **VRAM** (~8–24 GB depending on model/quantization), **16–32 GB+ system RAM**, **multi-GB model files on disk** (~4–40 GB each), plus **thermal/battery** load on a tablet — and, at tablet-feasible model sizes, **noticeably weaker quality** than frontier cloud models. It also strains the current "static web app, no backend, runs in a browser" design: in-browser local inference means **WebGPU (WebLLM/MLC)**, which is heavy and limited. Recommendation: treat onboard AI as a **separate future track with its own (much higher) spec sheet**; keep cloud as the baseline. **[→ Overview To Do — system-requirements section; onboard AI as a future track]**

9. **Future: support near-real-time and batched (asynchronous) communication too.** Beyond live face-to-face conversation, the system should eventually support **near-real-time and batched** channels — email, messaging, and stored text/speech (compose now, send/play later). This broadens the product from live conversation to the user's fuller communication life and connects to the Product Vision's framing (today's AAC users are *locked into* near-real-time/batch; we'd be supporting those modes *well*, by choice, rather than as a limitation). Future phase. **[→ Overview To Do]**

- TTS inflection: browser TTS has limited control (pitch, rate, volume on SpeechSynthesisUtterance). Natural-sounding placeholder delivery matters — monotone fillers will sound robotic. Voice banking / cloned voices (future) would give much better inflection control. Evaluate whether pitch/rate tweaks on fillers can improve naturalness in the near term.
  - **SSML does NOT work through the browser Web Speech API (finding, Ken's question June 16 2026).** The Windows built-in (SAPI/OneCore) and Google Chrome voices we use are driven via `SpeechSynthesisUtterance`, which has **no SSML support** — passing `<speak>`/`<prosody>`/`<break>` markup as the utterance text is read literally or stripped, not parsed (Chrome has never supported SSML here; don't depend on it). So inserting SSML "modifier characters" into a phrase to control inflection/timing is not an option with the current stack. **What actually works in-browser, three levers:** (1) **per-utterance** `pitch` (0–2), `rate` (0.1–10), `volume` (0–1) — set on the whole utterance, not mid-phrase; (2) **split a phrase into multiple chained utterances** each with different pitch/rate to fake an inflection contour (crude but the only way to vary prosody within a sentence); (3) **punctuation** — commas / periods / ellipses / em-dashes genuinely affect pause length and some intonation in most engines (this is the real "insert characters" answer), and `?`/`!` nudge intonation on some voices (engine-dependent, subtle). **Full SSML** (`<prosody rate/pitch>`, `<break time>`, `<emphasis>`, `<say-as>`) requires a **cloud TTS engine** — Google Cloud Text-to-Speech, Azure Speech, or Amazon Polly — which is a different architecture (network + per-character cost, user-funded like the LLM) and would plug into the same swappable TTS abstraction as voice banking. Near-term naturalness work is therefore limited to levers 1–3; richer control is the cloud-TTS / voice-banking future track.
  - **Reference doc:** `Shaping-Speech-Through-Text.docx` (generator `generate-tts-pronunciation-doc.js`) — practical per-character guide (punctuation for pacing/intonation, characters that change pronunciation, the per-voice caveat, a testing protocol via the "Speak my answer" button, and placeholder before/after candidates). Regenerable on request.
  - **EMPIRICAL FINDING (Ken tested, June 16 2026): punctuation/special characters for *prosody* are IGNORED by both the Google and Microsoft browser voices.** Commas/periods/ellipses/em-dashes did not produce usable pause or intonation changes on the voices we ship with — so **lever 3 (punctuation for pacing/intonation) is effectively a dead end** with the current stack. The doc's Section 3 is therefore over-optimistic for these engines and should be de-emphasized when next regenerated. **What remains usable at the text level is pronunciation, not prosody: phonetic respelling (and syllable hyphenation) of words the voice says wrong** — this is the lever Ken is investigating for proper nouns. Prosody control (pauses, pitch contour) is left to the code levers (per-utterance pitch/rate/volume, phrase-splitting) and, for real control, the cloud-TTS future track.
